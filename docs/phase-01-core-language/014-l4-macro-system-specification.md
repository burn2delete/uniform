## Purpose

L4 defines Gravity macros as compile-time transformations from syntax objects to syntax objects. Macros extend surface syntax without adding hidden semantics, hidden authority, or unchecked generated code.

The macro system exists because Gravity code is data. It remains safe because generated code is checked under the caller's profile, effects, capabilities, memory rules, and safety mode.

## Macro Model

A macro is a function that runs in a `:meta` context during compilation and returns syntax.

```clojure
(defmacro when [condition & body]
  `(if ~condition
     (do ~@body)
     nil))
```

The macro receives syntax objects, not raw lists. The result is syntax objects with generated-origin metadata. Expansion does not bypass `L2`, `L5`, `L6`, later `L15` provider checks, profile validation, or safety analysis.

## Syntax Object Contract

Macro input and output syntax objects contain:

- source form,
- source file and span,
- namespace context,
- lexical context,
- hygiene marks,
- metadata,
- compile phase,
- active caller profile,
- active macro namespace,
- generated-origin chain,
- build-effect context.

Macros may inspect syntax object structure and metadata. They may not mutate input syntax objects in place.

## Phase Separation

Macro execution happens at compile time. Runtime code generated by a macro executes later under the caller's profile.

Macro execution effects are build effects:

- `:build/read-file`,
- `:build/env`,
- `:build/network`,
- `:build/exec`,
- `:build/model-call`,
- `:build/tool-call`,
- `:compiler/read-ir`,
- `:compiler/write-ir`.

Hermetic builds deny network, shell, environment, model, and tool effects unless the project manifest grants them. A macro that needs those effects must declare them.

## Hygiene

Gravity macros are hygienic by default. Introduced bindings do not capture user bindings and user bindings do not accidentally capture introduced names.

Intentional capture must be explicit:

```clojure
(defmacro with-local-capture [name & body]
  (capture name
    `(let [~name 0]
       ~@body)))
```

The exact capture API is refined by macro implementation documents, but the invariant is fixed: accidental capture is a macro bug and must be detectable by fixtures.

## Expansion Rules

Expansion proceeds until no macro calls remain in macro positions or until the compiler reaches a configured recursion/depth limit.

Each expansion step records:

- macro identity,
- macro version,
- input syntax hash,
- output syntax hash,
- source span of call site,
- generated-origin links,
- build effects used,
- diagnostics emitted.

Macro expansion must preserve metadata unless the macro explicitly documents a metadata transformation.

Macro output must be ordinary Gravity syntax that lowers to core forms or declared domain IR boundaries.

## Profile and Safety Rules

A macro runs in `:meta`, but its output is checked in the caller's profile.

Examples:

- A macro used in `:kernel` cannot emit hosted reflection.
- A macro used in `:core` cannot emit filesystem access.
- A macro used in `:safe` code cannot emit unsafe islands unless the call site and policy explicitly allow them.
- A macro used in hermetic build mode cannot read environment variables unless granted.

Unsafe-generating macros must make unsafe behavior visible:

```clojure
(defmacro unchecked-nth [xs i]
  `(unsafe
     {:reason "unchecked indexed access generated by unchecked-nth"
      :source-span "generated:unchecked-nth:call-site"
      :profiles [:native :kernel]
      :effects [:memory/raw]
      :capabilities [:memory/raw]
      :preconditions [:index-in-bounds :vector-storage-stable]
      :postconditions [:element-returned :no-alias-created]
      :invariants [:no-safe-alias :storage-not-mutated-during-read]
      :safe-boundary unchecked-nth
      :evidence [:caller-index-proof-or-policy]
      :owner "macro-author"
      :review "MACRO-UNSAFE-UNCHECKED-NTH"
      :re-review :on-vector-layout-change}
     (vector/get-unchecked ~xs ~i)))
```

If policy disallows unsafe code, the expanded output is rejected with both macro call site and generated unsafe span.

## Macro Definitions

Macro definitions are namespace entries. They have:

- name,
- parameters,
- macro namespace,
- accepted syntax shape,
- build effects,
- required build capabilities,
- hygiene policy,
- output contract,
- version,
- documentation,
- fixtures.

Macros exported from a namespace are part of that namespace's public API.

## Macro Expansion Artifacts

Each compiled module emits a macro expansion trace:

```clojure
{:module app.main
 :expansions [{:macro gravity.core/when
               :call-span "src/app/main.gravity:12:3"
               :input-hash "sha256:..."
               :output-hash "sha256:..."
               :build-effects []
               :generated-spans ["generated:gravity.core/when:1"]}]
 :diagnostics []}
```

The trace is used by diagnostics, tooling, incremental compilation, conformance tests, and self-hosting equivalence.

## Requirements

- Macros must consume and produce syntax objects.
- Macro expansion must preserve source and generated-origin metadata.
- Macro build effects must be declared and checked against build policy.
- Macro output must pass normal type, effect, profile, capability, memory, and safety checks.
- Hygienic expansion is default; intentional capture is explicit.
- Expansion must be deterministic under the same source, macro versions, build grants, and dependency graph.
- Macro recursion and expansion size limits must produce stable diagnostics.

## Dependencies

L4 depends on `D0`, `D1`, `D3`, `D7`, `D8`, `L1`, `L2`, and `L3`.

It is upstream of type checking, effect checking, safety analysis, compiler syntax objects, formatter behavior, LSP macro expansion views, self-hosting, and macro safety.

## Outputs and Artifacts

L4 requires:

- macro namespace entries,
- macro build-effect records,
- macro expansion traces,
- generated-origin source maps,
- hygiene marks,
- macro diagnostics,
- macro conformance fixtures,
- macro version/provenance records.

## Rejected Behavior

L4 rejects:

- text substitution macros that claim syntax-object semantics,
- macros that drop source spans,
- macros that erase generated-origin chains,
- accidental capture,
- undeclared build effects,
- non-deterministic expansion without recorded inputs,
- macro output that bypasses caller profile checks,
- hidden unsafe code generated into safe contexts,
- expansion that depends on ambient filesystem, environment, shell, network, model, or tool authority.

## Diagnostics

- `L4-MACRO-NOT-SYNTAX`: macro returned non-syntax value.
- `L4-HYGIENE-CAPTURE`: expansion captures or is captured unintentionally.
- `L4-BUILD-EFFECT`: macro used undeclared or ungranted build effect.
- `L4-EXPANSION-DEPTH`: macro expansion exceeded configured depth.
- `L4-GENERATED-PROFILE`: generated code violates caller profile.
- `L4-GENERATED-UNSAFE`: macro generated unsafe code not allowed by policy.
- `L4-PROVENANCE-MISSING`: expansion output lacks generated-origin metadata.

## Conformance Criteria

- Fixtures cover simple macro expansion, nested expansion, syntax quote/unquote, metadata preservation, hygiene, intentional capture, and generated diagnostics.
- Hermetic build fixtures reject macros that read environment, network, shell, model, or tool state without grants.
- Profile fixtures show the same macro accepted in `:hosted` and rejected in `:kernel` when generated code is illegal.
- Unsafe macro fixtures produce unsafe audit artifacts or rejection diagnostics.
- Expansion traces are deterministic for the same inputs.
- Self-hosting fixtures compare seed and Gravity macro expansion traces.

## Change Control

Changing macro semantics affects every phase after syntax. Changes require updates to syntax object schema, macro fixtures, generated-origin diagnostics, formatter/LSP tooling, compiler bootstrapping, macro safety, and self-hosting equivalence tests.
