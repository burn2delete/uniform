# MATH9 - Symbolic Math and Rewrite System Specification

## Purpose

Gravity symbolic math is a guarded rewrite and proof system for numeric and
elementary expressions. It supports simplification, identity registration,
domain reasoning, EML normalization, candidate discovery, proof replay, and
counterexample reporting.

Symbolic rewriting is not string substitution. A rewrite is usable only when its
pattern, replacement, domain, branch policy, numeric mode, side conditions,
proof status, and invalidation rules match the expression being transformed.

## Requirements

- Every rewrite rule must state pattern, replacement, domain, codomain when
  relevant, branch policy, numeric modes, side conditions, proof status, and
  source provenance.
- Rewrites must preserve EFIR semantic anchors and generated-origin chains.
- Candidate identities from EML search start untrusted and cannot affect
  lowering until proof acceptance.
- Rewrites must not discard exceptional-value behavior, floating manifests,
  provider effects, runtime checks, or proof obligations.
- Rule application must emit a replayable trace.
- Rewrite engines must use deterministic ordering, fuel, or termination
  arguments to avoid unbounded simplification.
- Equality of symbolic trees, EFIR graphs, or EML trees is not semantic equality
  without a proof artifact.
- Equality saturation and e-graph extraction are compiler primitives only when
  their analyses, guards, costs, fuel, explanations, and proof replay artifacts
  are explicit.

## Rewrite Rule Form

```clojure
(defrewrite tan-via-sin-cos
  {:domain {x Real}
   :codomain Real
   :branch-policy {:tan :principal
                   :sin :real
                   :cos :real}
   :numeric-modes #{:symbolic :exact-real :interval}
   :side-conditions [(not= (cos x) 0)]
   :proof {:kind :symbolic
           :checker :gravity.rewrite/check
           :status :proved}
   :termination {:measure :operator-count
                 :direction :nonincreasing}}
  (tan x)
  (/ (sin x) (cos x)))
```

Rules may be declared by the standard library, packages, compiler providers, or
generated search tools. Package rules are imported under trust policy.

## Rule Artifact

```clojure
{:artifact :gravity/rewrite-rule
 :rule-id :tan-via-sin-cos
 :version 1
 :pattern '(tan x)
 :replacement '(/ (sin x) (cos x))
 :domain {x {:real :all}}
 :codomain {:real :all}
 :side-conditions ['(not= (cos x) 0)]
 :branch-policy {:tan :principal}
 :numeric-modes #{:symbolic :exact-real :interval}
 :proof-status :proved
 :proof-ref proof-hash
 :counterexamples []
 :invalidated-by #{:branch-policy-change :domain-change :mode-change}
 :source {:package :gravity.math
          :span source-span}}
```

The rule id and version are content-addressed over semantic fields.

## Rule States

Rules and identities use explicit states:

- `declared`: syntax is accepted but not proved.
- `candidate`: generated by EML search, symbolic search, or provider import.
- `guarded`: side conditions and domains are known but proof is pending.
- `proved`: exact equivalence accepted for declared modes.
- `bounded`: bounded-error replacement accepted through `MATH5`.
- `rejected`: counterexample, unsupported domain, branch mismatch, or proof
  failure.
- `deprecated`: rule remains readable for old artifacts but is not selected for
  new rewrites.

Only `proved` and `bounded` rules may change generated code.

## Side Conditions

Side conditions can mention:

- nonzero denominators,
- domain membership,
- branch policy,
- monotonic regions,
- periodic ranges,
- exceptional values,
- floating manifest constraints,
- mode constraints,
- provider-specific assumptions.

Side conditions are proof obligations. A rule application that cannot discharge
them becomes a candidate trace entry, not an accepted rewrite.

## Rewrite Engine

The engine operates in phases:

1. import EFIR or EML expression and semantic context,
2. select rules allowed by profile, mode, branch policy, and trust policy,
3. match patterns with typed variables,
4. check side conditions or emit proof requests,
5. apply accepted rewrites,
6. record trace entries,
7. update proof obligations and invalidated facts,
8. run verifier replay.

Rule ordering is deterministic. If heuristic cost search is used, the emitted
trace must include the cost model, candidate ordering, fuel, selected candidate,
and rejected candidates that were relevant to the choice.

## E-Graph Equality Saturation

Equality saturation is a first-class rewrite strategy, not an incidental
optimizer detail. An e-graph may group equivalent expressions only under the
domain, branch policy, numeric mode, floating manifest, provider effect, and
side-condition guards that justified each equality edge.

An e-class analysis records facts that are valid for every member selected under
the current guard set. Required analyses include:

- domain and codomain facts,
- branch-policy facts,
- numeric mode and floating manifest facts,
- exceptional-value behavior,
- interval or monotonic facts when used by a rule,
- provider effect and runtime-check preservation,
- proof status of equality edges,
- source and generated-origin coverage.

Merging e-classes joins analysis facts conservatively. If facts conflict, the
merge is rejected, guarded to the intersection where it is proved, or recorded
as a candidate edge that cannot be extracted as an accepted rewrite.

Saturation must be bounded. An e-graph run declares deterministic limits for
iterations, node count, e-class count, rule firings, per-rule fuel, and proof
requests. Exhaustion emits a saturation report and explanation trace. It does
not make the current best expression semantically valid unless extraction can be
proved from accepted edges.

Extraction uses an explicit, versioned cost model. The model may score operator
count, expected runtime, numeric stability, allocation, provider calls, runtime
checks, source stability, proof availability, and generated-code size. It must
define deterministic tie breaks and must not hide a semantic requirement inside
an implementation heuristic.

```clojure
{:artifact :gravity/egraph-extraction
 :run-id saturation-hash
 :root-eclass :e17
 :selected expr-b
 :cost-model {:id :math9-default-cost
              :version 3
              :weights {:operator-count 1
                        :runtime-check 4
                        :unproved-edge :forbidden}
              :tie-break [:proof-depth :source-order :node-id]}
 :bounds {:iterations 40
          :nodes 10000
          :per-rule-fuel 128}
 :guards {:domain {x {:real :all}}
          :branch-policy {:tan :principal}}
 :proof-replay [trace-id-1 trace-id-2]
 :explanation explanation-hash
 :status :proved}
```

Proof replay for extraction reconstructs the selected expression from the input
root through rule applications, e-class merges, and side-condition results. It
must be possible without rerunning saturation, rescheduling rules, or trusting
the extractor. Explanation traces include the selected path, e-class ids,
analysis facts used by extraction, rejected candidate representatives that
affected the cost decision, exhausted fuel, and all guards attached to equality
edges.

Unproved edges may exist in an e-graph for discovery and diagnostics. They may
not be used to extract accepted code, simplify a proof obligation, discharge a
side condition, or claim semantic equality. If extraction would require an
unproved edge, the result is a candidate trace entry with a diagnostic, not an
accepted rewrite.

## Trace Entry

```clojure
{:trace-id trace-hash
 :step 9
 :rule :tan-via-sin-cos
 :state :proved
 :before expr-a
 :after expr-b
 :matched {:x node-id}
 :side-condition-results [{:condition '(not= (cos x) 0)
                           :result :proved
                           :proof proof-id}]
 :preserved #{:domain :branch-policy :floating-manifest}
 :invalidated #{:cost-estimate}
 :source {:efir-node :n14
          :span source-span}}
```

Verifier replay must be possible without running the optimizer that selected the
rewrite.

## Termination and Fuel

A rewrite set must provide one of:

- a decreasing measure,
- a stratified phase order,
- bounded fuel with deterministic exhaustion behavior,
- e-graph extraction with a bounded cost model and proof replay,
- proof-engine-guided application that emits a finite trace.

Fuel exhaustion is not a correctness failure by itself. It means no accepted
rewrite beyond the recorded trace was selected.

## Counterexamples

Counterexample artifacts record rejected rules or rejected applications:

```clojure
{:rule :sqrt-square
 :claim '(= (sqrt (* x x)) x)
 :domain {x {:real :all}}
 :counterexample {x -1}
 :reason :domain-too-wide
 :suggested-guard '(>= x 0)}
```

Counterexamples are conformance fixtures. A later narrower rule may be accepted,
but the broad rejected claim remains invalid.

## Diagnostics

Symbolic rewrite diagnostics use `MATH9` identifiers:

- `MATH9-RULE-SHAPE` for malformed rules.
- `MATH9-DOMAIN` for missing or incompatible domains.
- `MATH9-BRANCH` for branch-policy mismatch.
- `MATH9-SIDE-CONDITION` for undischarged side conditions.
- `MATH9-PROOF` for unproved rules used as accepted rewrites.
- `MATH9-TRACE` for unreplayable rewrite traces.
- `MATH9-TERMINATION` for unbounded rewrite strategy.
- `MATH9-COUNTEREXAMPLE` for identities disproved by fixtures or proof tools.
- `MATH9-EGRAPH` for invalid e-class analysis, guard, fuel, cost, or
  explanation artifacts.
- `MATH9-EQUALITY` for tree identity, e-class membership, or unproved extraction
  used as semantic equality.

Diagnostics must include rule id, rule version, expression id, EFIR graph id,
mode, domain, branch policy, source span, proof status, and remediation.

## Rejected Designs

Gravity rejects global algebraic identities without domain and branch guards.

Gravity rejects EML, EFIR, or symbolic tree identity as equality evidence.

Gravity rejects rewrite systems with unbounded nondeterministic application.

Gravity rejects e-graph extraction that depends on unproved equality edges.

Gravity rejects cost models that choose a representative without replayable
guarded proof.

Gravity rejects rewrites that erase floating manifests or exceptional behavior.

Gravity rejects imported package rules accepted solely by package trust.

## Conformance Criteria

A conforming symbolic rewrite implementation must demonstrate:

- rule declaration, import, and versioning,
- accepted exact rewrite with proof replay,
- accepted bounded rewrite with certificate reference,
- rejected rewrite with counterexample,
- side-condition proof and failure cases,
- branch-sensitive rewrite rejection,
- deterministic trace replay,
- fuel or termination behavior,
- bounded equality-saturation run with e-class analysis summary,
- extraction cost model with deterministic tie breaks,
- e-graph explanation trace and proof replay without rerunning saturation,
- rejection of extraction that requires an unproved equality edge,
- domain-guarded e-class merge and conflicting-analysis rejection,
- diagnostics for unproved, unbounded, and tree-equality cases.
