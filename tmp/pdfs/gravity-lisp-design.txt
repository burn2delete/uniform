Gravity: a Lisp for the Whole Stack
1. Core thesis
Gravity should not try to be “one runtime everywhere.” That fails because bare-metal
firmware, kernels, browsers, cloud services, distributed workflows, and AI agents have
incompatible runtime assumptions.
The correct design is:
One homoiconic language, one semantic model, many compilation profiles.


===== PAGE 2 =====
Gravity is a Lisp/Clojure-inspired language whose source code is data, whose compiler is
programmable from inside the language, and whose semantics are stratified so the same
language can target:
hardware descriptions

The language should be mostly self-hosted: reader, macroexpander, type/effect checker,
optimizer, package system, build system, and standard library written in Gravity. Only a tiny

2. Design principles
Homoiconic first
Gravity code is structured data.

code generators operate on explicit syntax values.
Clojure-inspired, but not Clojure-compatible
Gravity should borrow from Clojure:
persistent data structures

multimethod-like dispatch

One language, many profiles
Every Gravity namespace declares a profile.

A kernel module cannot use dynamic eval, GC, unbounded allocation, reflection, network

Same language. Different compile-time contracts.

A function does not just have a type. It has a type plus effects.

:- User
...)
Possible effects:
clojure
:pure
:alloc
:io/read

For AI workflows, nondeterminism is explicit:

8. Memory model
Gravity needs multiple memory regimes.
Managed memory


===== PAGE 10 =====
Used in hosted/cloud/app code.
clojure
(def users [{:id 1 :name "Ada"}
{:id 2 :name "Rich"}])
Backed by persistent data structures and GC.
Region memory
Useful for high-performance native code.
clojure
(with-region [r]
(let [buf (region/alloc r Byte 4096)]
(process-buffer buf)))
Region memory is freed at the end of the lexical scope.
Linear resources
Useful for files, sockets, locks, buffers, GPU handles, device memory.
clojure
(with-open [file (fs/open "/tmp/data.bin")]
(fs/read-all file))
The compiler ensures the resource is consumed exactly once.
Raw memory
Only available in unsafe/system profiles.
clojure
(unsafe
(ptr/store! address value))
Unsafe code is not banned, but it is syntactically and semantically isolated.

Gravity packages should declare capabilities.

15. Self-hosting strategy
Gravity should bootstrap in stages.
Stage 0: Seed compiler
Write a tiny compiler in Clojure, Rust, or C.
Responsibilities:
read Gravity forms
expand minimal macros
compile core subset
emit C, JVM bytecode, or LLVM IR
This stage should be intentionally disposable.
Stage 1: Gravity reader and macroexpander
Rewrite the reader, syntax model, and macroexpander in Gravity.
clojure
(ns gravity.reader
(:profile :core))
Stage 2: Gravity analyzer
Write the AST analyzer, resolver, namespace system, and type/effect checker in Gravity.
clojure
(ns gravity.analyzer
(:profile :meta))
Stage 3: Gravity MIR
Define Gravity MIR as ordinary Gravity data.
clojure
{:op :call
:fn 'gravity.core/+
:args [{:op :local :name 'a}
{:op :local :name 'b}]
:type 'I64
:effects #{}}

27. The important philosophical split
Gravity should have three layers:
1. Gravity the language
The Lisp syntax, core semantics, type/effect system, macros, protocols, namespaces.
2. Gravity the compiler platform
The self-hosted compiler, IRs, passes, backends, package/build system.
3. Gravity the execution ecosystem
Runtimes for native, hosted, distributed, AI, hardware, and OS-level targets.
Trying to make these one thing will make the design incoherent.

:unsafe No check, unsafe only

Mode Behavior

Example:
clojure

The compiler can prove the access is in bounds and remove the check.
Safe source. Fast binary.
10. Numeric safety
Many vulnerabilities are not memory bugs directly. Integer overflow, truncation, and invalid
casts can become security bugs.
Gravity should avoid C-style silent numeric hazards.
Default arithmetic
In safe mode:
clojure
(+ a b)
should be checked for overflow for fixed-width integers unless the compiler proves it safe.
Explicit alternatives:
clojure
(wrapping/+ a b)
(saturating/+ a b)
(checked/+ a b)
(unchecked/+ a b) ;; unsafe or performance-profile restricted
Casts should be explicit:


===== PAGE 100 =====
clojure
(cast/u8 x) ;; checked
(cast/u8-truncate x) ;; explicit truncation
No implicit narrowing.
11. Data-race safety
Gravity should guarantee data-race freedom in safe concurrent code.
Safe shared mutation requires one of:
atomics
locks
STM
actors
channels
ownership transfer
immutable persistent data
Example actor:
clojure
(defactor Counter
(state [n 0])
(on :inc []
(update state :n inc)))
Example ownership transfer:
clojure
(task/spawn
(move buffer)
(process buffer))
After moving buffer into the task, the parent cannot use it.
This avoids shared mutable aliasing across threads.

12. Capability safety
Memory safety is not enough. A safe language can still delete files, exfiltrate data, or let an
AI agent call tools it should not call.
Gravity already has effects/capabilities. Safety should integrate directly with that.
clojure
(defn delete-user-data
{:effects #{:filesystem/write}}
[path]
(fs/delete path))
A deployment must grant the capability:
clojure
(deployment
{:allow #{:filesystem/read}
:deny #{:filesystem/write}})
Then delete-user-data cannot run there.
This matters for:
file access
network access
environment variables
secrets
shell execution
database writes
model calls
tool calls
FFI
raw memory
unsafe compiler plugins
13. FFI safety
Foreign function interfaces are one of the largest safety holes in safe languages.
Gravity should treat FFI as unsafe by default.


===== PAGE 102 =====
clojure
(extern strlen
{:link "strlen"
:type (Fn [CString] USize)
:effects #{:ffi}
:safety :unsafe})
To make FFI safe, a wrapper must declare and prove or validate contracts.
clojure
(defn safe-strlen
{:safe-wrapper-for strlen
:requires [(cstring/null-terminated? s)]
:ensures [(<= result (cstring/max-length s))]}
[s :- CString]
:- USize
(unsafe
(strlen s)))
Safe code imports the wrapper, not the raw function.
14. Unsafe islands
Unsafe should be syntactically obvious, semantically tracked, and reviewable.
clojure
(unsafe
{:reason "Volatile read from hardware register"
:effects #{:memory/mmio}
:invariants [(aligned? addr 4)
(valid-mmio-region? addr)]}
(mmio/read32 addr))
Unsafe blocks should produce compiler artifacts:
clojure
{:unsafe-block/id "drivers.uart/read-status:42"
:reason "Volatile read from hardware register"
:effects #{:memory/mmio}
:invariants [...]
:review-status :required}
Then the build can enforce:


===== PAGE 103 =====
clojure
(:safety {:unsafe-policy :deny})
or:
clojure
(:safety {:unsafe-policy :allow-reviewed-only})
This is essential for enterprise, kernel, medical, aerospace, finance, and security-critical
use.
15. Safety levels

:trusted-runtime Runtime/compiler internals can use unsafe
:experimental Relaxed checks for research
:unsafe No guarantees; useful only for low-level escape hatches

Gravity must allow high-performance internals while exposing safe APIs.

24. Safety and performance should not be enemies
Gravity should avoid the false choice between safety and performance.
Compiler strategy:
Start safe.
Prove checks unnecessary.
Erase proven checks.
Keep only necessary checks.
Example:
clojure
(loop [i 0]
(when (< i n)
(use (vector/get xs i))
(recur (inc i))))
If the compiler proves:
clojure
(<= n (count xs))
then bounds checks can be removed.
Safety remains semantic. Runtime overhead disappears where proven unnecessary.
This should be a core optimization principle:
Safety checks are specifications first and runtime operations only when proof is
unavailable.
